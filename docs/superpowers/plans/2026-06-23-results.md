## HONEST verdict: partial win, core mechanism NOT yet working

**What improved (real):** treatment cut input tokens and cost ~43-45% and roughly
halved the loop-inflation share (53%→28% of input). The other components (prime
gate, loop counting, prefix-stable, measurement) are live and not harmful.

**What FAILED (the headline goal D):** the map-reduce "agent-in-agent" did NOT
activate. Probing the running reasonix transcript: `run_skill` / `spawn_subagent`
/ `map-reduce` appear **0 times** — reasonix never invoked the skill, so synthesize
still looped and never produced a real report.

**Root cause (verified by directly probing reasonix acp):**
1. reasonix acp DOES have a `run_skill` tool AND DOES see the `map-reduce-synthesis`
   skill in its pinned index (`[🧬 subagent]`). So loading is fine.
2. BUT reasonix has **NO `spawn_subagent` tool** — the skill body I wrote instructs
   it to call `spawn_subagent`, which does not exist. The reasonix subagent
   mechanism is: `run_skill` invokes a `runAs:subagent` skill (the subagentRunner
   spawns the isolated child). So a map-reduce skill must spawn its workers by
   calling `run_skill` on ANOTHER `runAs:subagent` skill — not `spawn_subagent`.
   And a subagent skill receives ONLY its `arguments` as context.
3. Even with a correct skill, reasonix-flash did not pick the skill from a prompt
   directive on its own — the gateway likely must make the activation stronger
   (or the skill must be the single obvious path).

## Next (to make D actually work)
- Rewrite the skill to use reasonix's real mechanism: a `map-reduce-synthesis`
  orchestrator skill (runAs:subagent) that spawns workers via `run_skill` on a
  small `synthesis-map-worker` skill (also runAs:subagent), each fed its group via
  `arguments`; a final `run_skill` merge step. Verify reasonix allows that nesting
  depth (one subagent calling run_skill on another) — if not, do Map sequentially
  inside the single orchestrator turn (still bounded, no oversized one-shot).
- Strengthen activation: either make the gateway directive unmistakable, or
  consider gateway-side map-reduce (split the claim block itself before sending)
  as the deterministic fallback that does not depend on the model choosing a skill.
- Re-run the same A/B and confirm synthesize COMPLETES (real report) with the
  loop-inflation bucket near zero.

Components A/B/C/E are committed and live (8b7a8b4, b5d849c). Component D is built
but proven non-functional end-to-end; it needs the skill-mechanism rewrite above
before it can be claimed.

## UPDATE — Component D FIXED (2026-06-23, second session)

Root causes from the first e2e run were addressed:
1. **No `spawn_subagent` tool in reasonix** → rewrote the skill to do Map/Reduce as
   small SEQUENTIAL steps inside its own subagent loop (no nested spawn).
2. **flash ignored the soft directive** → the gateway directive is now a
   `MANDATORY FIRST ACTION` block. Direct acp probe: reasonix now CALLS run_skill.
3. **flash over-thought the skill** (17+ min, 107KB reasoning) → added HARD LIMITS
   to the skill (≤4 batches, ~5 terse steps, JSON-only). Probe dropped to **16s**.
